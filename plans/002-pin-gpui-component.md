## Why this matters

`gpui` is pinned to an exact Zed rev with a comment explaining that the whole
tree must resolve to a single gpui. But `gpui-component` and
`gpui-component-assets` — from the same upstream repo — float on the default
branch HEAD. Today `Cargo.lock` holds them at a working commit, but any
`cargo update` (or a fresh resolve without the lockfile) pulls whatever
upstream merged last, which can break the build or silently change UI
behavior. Pinning both to the currently-locked rev makes builds reproducible
and makes upgrades a deliberate act.

## Steps

### Step 1: Add `rev` to both gpui-component dependencies

In `Cargo.toml`, change the two entries to:

```toml
# Pin gpui-component to the rev the lockfile was tested at. Bump this together
# with the gpui rev above (they must agree on a single gpui).
gpui-component = { git = "https://github.com/longbridge/gpui-component", rev = "063e55bbc4fb13907a988111e3581595cbcaefde", features = [
  "tree-sitter-markdown",
] }
gpui-component-assets = { git = "https://github.com/longbridge/gpui-component", rev = "063e55bbc4fb13907a988111e3581595cbcaefde" }
```

(First confirm `063e55bb…` is still the rev in `Cargo.lock` — use the grep
command above. If the lockfile shows a different hash, use that one instead.)

**Verify**: `cargo check --all-targets` → exit 0.

### Step 2: Confirm the lockfile kept the same commit

**Verify**: `git diff Cargo.lock` → the only changes are the `source = ` URLs
gaining `?rev=…`; the commit hash after `#` is unchanged. If the hash changed,
STOP (see below).
